<p class=MsoNormal>Hi All,<br>
<br>
Some time ago someone reported an strange memory behavior in mac os x on the
gecode list. I do not know whether it was fixed but i think it still remains in
the current implementation. The problem apparently happens only in mac os x.
You can get it by replacing the main function of the queens example to this
one:<br>
<br>
int<br>
main(int argc, char* argv[]) {<br>
&nbsp; <br>
&nbsp; SizeOptions opt(&quot;Queens&quot;);<br>
&nbsp; opt.iterations(500);<br>
&nbsp; opt.size(100);<br>
&nbsp; opt.propagation(Queens::PROP_<o:p></o:p></p>

<div>

<p class=MsoNormal>DISTINCT);<br>
&nbsp; opt.propagation(Queens::PROP_BINARY, &quot;binary&quot;,<br>
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;
&quot;only binary disequality constraints&quot;);<br>
&nbsp; opt.propagation(Queens::PROP_MIXED, &quot;mixed&quot;,<br>
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;
&quot;single distinct and binary disequality constraints&quot;);<br>
&nbsp; opt.propagation(Queens::PROP_DISTINCT, &quot;distinct&quot;,<br>
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;
&quot;three distinct constraints&quot;);<br>
&nbsp; opt.parse(argc,argv);<br>
&nbsp; /*<br>
&nbsp; Example::run&lt;Queens,DFS,SizeOptions&gt;(opt);<br>
&nbsp; */<br>
&nbsp; std::vector&lt;Queens*&gt; q;<br>
&nbsp; for (unsigned int i=0; i &lt; 100000; i++) {<br>
&nbsp;&nbsp;&nbsp; q.push_back(new Queens(opt));<br>
&nbsp; }<br>
&nbsp; std::cout &lt;&lt; &quot;finished insertion&quot; &lt;&lt; std::endl;<br>
<br>
&nbsp; for (unsigned int i=0; i &lt; 100000; i++) {<br>
&nbsp;&nbsp;&nbsp; Queens *tmp = q[i];<br>
&nbsp;&nbsp;&nbsp; delete tmp;<br>
&nbsp;&nbsp;&nbsp; q.pop_back();<br>
&nbsp; }<br>
&nbsp; std::cout &lt;&lt; &quot;finished delete&quot; &lt;&lt; std::endl;<br>
<br>
&nbsp; int a;<br>
&nbsp; std::cin &gt;&gt; a;<br>
&nbsp; return 0;<br>
}<br>
<br>
The idea is to create a lot of spaces and to keep pointer to them in an array.
After the creation all spaces are deleted. The expected behavior is to release
all the memory allocated in the first loop and to have it available by the
operating system. I tried in a mac os x running leopard and the memory seems to
be still used by the process (according to Activity Monitor). In a linux
machine the memory seems to be released according to top.<br>
<br>
Another thing is that when using the gecode trunk to try the same problem, the
de-allocation behaves better, that is, half of the memory is released but still
some memory is kept by the process.<br>
<br>
Do you have any suggestion on how to work around this?<o:p></o:p></p>
