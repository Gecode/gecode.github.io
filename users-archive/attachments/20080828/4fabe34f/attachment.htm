<p class=MsoNormal><span style='font-size:11.0pt;font-family:"Calibri","sans-serif";
color:#1F497D'>If you increase hcsz_min to 16 * 1024 that would avoid part of
the problem. One part are the spaces which are very small and few. But per
space you will typically allocate a number of heap chunks. Initially these
chunks will be small and will only dynamically grow when demand justifies it.
So, if you increase, you'll have more slack but more blocks will be returned to
the OS.<o:p></o:p></span></p>

<p class=MsoNormal>That's unlikely. &nbsp;The problem is that the memory
footprint is going to be the maximum memory used by spaces that are alive at
the same time. &nbsp;In your example, that's tens of thousands of spaces.
&nbsp;During search, it's never more than a few spaces (search tree depth /
recomputation depth). &nbsp;If the tree depth is very big, you're lost anyway
as the search is not going to find a solution.<o:p></o:p></p>
