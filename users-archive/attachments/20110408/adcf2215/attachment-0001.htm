<div>&gt;&gt; 2) My executables, even after dead code elimination and other optimisations are very large. It's around 17 Mb. I'm not sure there's an easy fix for this, but any ideas would be welcome, for how to reduce file size of the executable. It seems that there are many features I'm not going to need, so perhaps I can try to strip down the library.<br>



&gt;<br>
&gt; First of all, that sounds like you're linking statically, right? &nbsp;It's true, the Gecode libraries are a bit on the fat side. &nbsp;What you can definitely do is strip the debug symbols. &nbsp;Other than that, I think the linker already only includes the symbols that it needs, so there's not much you can do in addition.<br>



<br>
</div>Yes, statically. I'm not sure there's any other way when it comes to building iPhone apps. That 17 Mb is for a "fat" binary that supports two architectures (armv6 and armv7). So the real figure is about half that. Still it would be nice if it was smaller.</blockquote>


</div><div><br></div>Stripping debug symbols helps a lot with size. On my machine, the size of the dynamic libraries goes from 52 MiB to 7.1 MiB when stripped.<div><br></div><div>In addition, you can try and compile Gecode with -Os as optimization flag. It won't be as fast, but it might make the executable smaller (I haven't tried it, you'll need to experiment). Also, make sure that you are only including the parts that you need (Gist and flatzinc might not be relevant for example).&nbsp;</div></blockquote><div><br></div><div>Gist won't be compiled anyway (no Qt on iOS), and flatzinc is usually not linked unless you actually use it.</div><div><br></div><div>On Darwin&nbsp;(Mac OS, iOS), the option for minimizing size is -Oz (-Os also exists but is slightly less radical). &nbsp;Here's a table of executable sizes (x86_64 on Mac OS):</div><div><br></div><div><span class="Apple-tab-span" style="white-space:pre">                                        </span>-O3<span class="Apple-tab-span" style="white-space:pre">                </span>-O3 stripped<span class="Apple-tab-span" style="white-space:pre">                </span>-Oz<span class="Apple-tab-span" style="white-space:pre">                </span>-Oz stripped<span class="Apple-tab-span" style="white-space:pre">        </span>-Oz clang<span class="Apple-tab-span" style="white-space:pre">        </span>-Oz clang stripped</div><div>examples/queens:<span class="Apple-tab-span" style="white-space:pre">        </span>8,2M<span class="Apple-tab-span" style="white-space:pre">        </span>4,9M<span class="Apple-tab-span" style="white-space:pre">                        </span>7,6M<span class="Apple-tab-span" style="white-space:pre">        </span>3,8M<span class="Apple-tab-span" style="white-space:pre">                </span>7,3M<span class="Apple-tab-span" style="white-space:pre">                </span>3,3M</div><div>tools/flatzinc/fz:<span class="Apple-tab-span" style="white-space:pre">                </span>15M<span class= <div>If you really want to make the executables as small as possible, then you could start to remove parts that you are not using. You might want to investigate if there is a way to make your compilation tool-chain do automatic dead-code removal.</div></blockquote><div><br></div><div>I thought the linker did that automatically, but apparently, it doesn't. &nbsp;You can tell the linker explicitly to remove dead code. &nbsp;On Darwin, that's done using the -dead_strip flag (and that's possibly enabled by default in XCode, I'm not sure). &nbsp;</div><div>With -dead_strip, examples/queens goes down to 4,4M with -Oz, or 2,3M stripped. &nbsp;So, comparing -O3 nonstripped to -Oz -Wl,-dead_strip stripped, we can reduce the binary by 70%.</div><div><br></div><div>Still, quite a lot of symbols end up in the executable that I can't explain, e.g. most of the set library seems to be included in the queens executable. &nbsp;So indeed, if your constraint model does not use set constraints, consider compiling Gecode with --disable-set-vars. &nbsp;Here are the numbers for queens without set variables:</div><div><div><span class="Apple-tab-span" style="white-space: pre; ">                                        </span>-Oz<span class="Apple-tab-span" style="white-space: pre; ">                </span>-Oz stripped <span class="Apple-tab-span" style="white-space:pre">        </span>-Oz -dead_strip <span class="Apple-tab-span" style="white-space:pre">        </span>-Oz -dead_strip stripped</div><div>examples/queens:<span class="Apple-tab-span" style="white-space: pre; ">        </span>6,2M<span class="Apple-tab-span" style="white-space: pre; ">        </span>2,9M<span class="Apple-tab-span" style="white-space: pre; ">                </span>3,2M<span class="Apple-tab-span" style="white-space: pre; ">                        </span>1,5M</div><div><br></div><div>We could probably reduce the size even further by carefully looking at the symbols and removing unused stuff. &nbsp;But anyway, this already saves 80% code size.</div><div><br></div><div>Another thing I tried (just for fun) was to compile using clang with -O4, enabling link-time optimization. &nbsp;The smallest size I could get for queens (but including set variables) was 2,2M, which is not too bad either, considering that this does full optimization.</div></div><div><br></div><div>Cheers,</div><div><span class="Apple-tab-span" style="white-space:pre">        </span>Guido</div><div><br></div></div><div>
